# LLM Matching Template

## Category intent

`matching`: map requirements to profile evidence and propose claim coverage.

Primary output is structured mapping JSON plus deterministic review surfaces.

## Entries (required)

- translated job understanding (`translate/approved/state.json`),
- profile/base knowledge input,
- optional admissibility policy constraints.

Example entry contract:

```python
class MatchingInput(BaseModel):
    job_id: str
    requirements: list[dict]
    profile_evidence: list[dict]
    policy: dict
```

## Outputs

- `nodes/match/proposed/state.json`
- `nodes/match/proposed/view.md` (generated by `sync_json_md`)
- `nodes/match/review/decision.md` (generated by `sync_json_md`)

Review surfaces:

- always `generate_review_surface=True`.

## `logic.py` structure

```python
class MatchProposal(BaseModel):
    schema_version: str
    doc_type: str
    mappings: list[dict]
    gaps: list[dict]
    summary: dict


def run_logic(*, runtime: LLMRuntime, prompt_bundle: PromptBundle, logic_input: MatchingInput) -> MatchProposal:
    ...
```

## Prompt management

- `prompt/system.md`: admissibility and evidence-linking policy.
- `prompt/user_template.md`: requirement + profile slots.
- `prompt/output_schema.json`: strict mapping schema.
- `prompt/meta.yaml`: prompt identity/version.
- optional `fewshots.jsonl`: mapping examples.

Prompt rules:

1. every proposed claim must reference evidence ids,
2. no claim outside approved profile evidence,
3. keep machine fields complete for review parser compatibility.

## Node routing expectations

- returns `status: "pending_review"` on success,
- deterministic review node owns approval authority.

## Anti-wax checks

1. no fixed generic mappings reused across jobs,
2. no empty mapping list with success status,
3. no fabricated evidence ids.
