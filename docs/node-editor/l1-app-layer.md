# L1 — App Layer

> The orchestrator. Thin by design: fetch data, translate to AST, mount the editor.

---

## Responsibilities

| What | File |
|------|------|
| App routing | `app/App.tsx` |
| Global nav shell | `components/layouts/AppShell.tsx` |
| Fetch + translate + mount editor | `features/graph-editor/L1-app/GraphEditorPage.tsx` |
| Domain → AST translation | `features/graph-editor/lib/schema-to-graph.ts` |
| AST → domain serialization | `features/graph-editor/lib/graph-to-domain.ts` |

## What L1 does NOT contain

Sidebar, filters, undo, save, inspectors, creation palette. Those belong to the editor (L2) and travel with it.

```typescript
export function GraphEditorPage() {
  const { data: rawData, isLoading } = useQuery({
    queryKey: ['graph'],
    queryFn: () => mockClient.getGraph(),

  const saveMutation = useMutation({
    mutationFn: (ast: { nodes: ASTNode[]; edges: ASTEdge[] }) =>
      mockClient.saveGraph(graphToDomain(ast.nodes, ast.edges)),
  });

  if (isLoading) return <LoadingSkeleton />;

  const { nodes, edges } = schemaToGraph(rawData, schema);

  return (
    <GraphEditor
      initialNodes={nodes}
      initialEdges={edges}
      onSave={(n, e) => saveMutation.mutate({ nodes: n, edges: e })}
    />
  );
}
```

L1 knows about the domain ("graph data"), calls `schemaToGraph`, and passes typed props to L2. It does not know what ReactFlow is.

## schemaToGraph Pipeline

The translation from domain data to canvas-ready AST runs in 4 phases. Each phase is independently testable.

```typescript
// features/graph-editor/lib/schema-to-graph.ts

/** Phase 1: Match raw data items against node type rules */
function matchNodes(rawData: unknown, nodeTypes: NodeTypeRule[]): MatchedNode[];

/** Phase 2: Resolve parent/child topology (groups, subflows) */
function resolveTopology(matched: MatchedNode[], rules: TopologyRule[]): TopologyGraph;

/** Phase 3: Generate edges from array fields and external references */
function resolveEdges(topology: TopologyGraph, edgeTypes: EdgeTypeRule[]): RawAST;

/** Phase 4: Validate each payload against the Node Type Registry Zod schemas */
function validateAST(ast: RawAST, registry: NodeTypeRegistry): {
  nodes: ASTNode[];           // valid nodes
  edges: ASTEdge[];           // valid edges
  errors: ValidationError[];  // invalid nodes → rendered as ErrorNode, no crash
};

export function schemaToGraph(rawData: unknown, schema: Schema): ValidatedAST {
  const matched = matchNodes(rawData, schema.nodeTypes);
  const topology = resolveTopology(matched, schema.topology);
  const rawAst = resolveEdges(topology, schema.edgeTypes);
  return validateAST(rawAst, registry);
}
```

### What each phase does

**Phase 1 — matchNodes**

Iterates the raw domain data and applies `match_rule` from the schema to assign a `typeId` to each item.

Example schema rule:
```json
{
  "match_rule": { "property": "nivel", "value": 3 },
  "type_id": "vehicle",
  "render_as": "node",
  "color_token": "token-vehicle"
}
```

**Phase 2 — resolveTopology**

Reads `render_as: "group"` items and builds parent/child relationships via `parentId`. These become RF subflows.

**Phase 3 — resolveEdges**

Iterates array fields (e.g. `conexiones_externas: ["ext_motor"]`) and generates `ASTEdge` objects.

**Phase 4 — validateAST**

Calls `registry.validatePayload(typeId, payload)` for each node. Invalid payloads produce `ErrorNode` renderings — no runtime crash.

### Why 4 phases (not 1 monolithic function)

The old approach mixed all three concerns:
- "What type is this?" (matching)
- "Where does it go?" (topology)
- "What connects to what?" (edges)

Separating them makes each phase unit-testable:
```typescript
// test: only test phase 2 with a pre-built MatchedNode[]
const topology = resolveTopology(mockMatchedNodes, rules);
expect(topology.nodes.find(n => n.id === 'heli').parentId).toBe('aereos');
```

## AST Shape

What the canvas receives — no domain knowledge, only topology and visual tokens:

```typescript
interface ASTNode {
  id: string;
  typeId: string;              // matches a registry entry
  parentId?: string;           // enables subflows
  visualToken: string;         // color token → xy-theme.css
  payload: unknown;            // validated by Zod in registry
  position?: { x: number; y: number };
}

interface ASTEdge {
  id: string;
  source: string;
  target: string;
  typeId: string;
  visualToken: string;
  label?: string;
}
```

The canvas knows nothing about "Vehicles" or "Jobs". It only sees `typeId`, `parentId`, and `visualToken`.

## Schema DSL

The schema JSON maps domain structure to visual behavior:

```json
{
  "nodeTypes": [
    {
      "matchRule": { "property": "nivel", "value": 1 },
      "typeId": "categoria-raiz",
      "renderAs": "group",
      "colorToken": "token-grupo-raiz"
    },
    {
      "matchRule": { "property": "nivel", "value": 3 },
      "typeId": "modelo-vehiculo",
      "renderAs": "node",
      "colorToken": "token-vehiculo"
    }
  ],
  "edgeTypes": [
    {
      "relation": "usa_componente",
      "source": "modelo-vehiculo",
      "targetArray": "conexiones_externas",
      "colorToken": "token-edge-dependencia"
    }
  ]
}
```

Each `typeId` must have a corresponding entry in the Node Type Registry (see `l2-node-type-registry.md`).

## Testing L1

Each pipeline phase is independently testable with unit tests:

```typescript
// Unit test: matchNodes
test('nivel=3 nodes match as modelo-vehiculo', () => {
  const result = matchNodes(rawData, schema.nodeTypes);
  expect(result.find(n => n.nivel === 3)?.typeId).toBe('modelo-vehiculo');
});

// Unit test: validateAST produces ErrorNode for malformed payload
test('invalid payload → ErrorNode, no throw', () => {
  const { nodes, errors } = validateAST(invalidAst, registry);
  expect(errors).toHaveLength(1);
  expect(nodes.find(n => n.typeId === 'error')).toBeDefined();
});

// Contract test: L1 output is consumable by L2
test('schemaToGraph output satisfies GraphEditorProps schema', () => {
  const { nodes, edges } = schemaToGraph(rawData, schema);
  expect(() => graphEditorPropsSchema.parse({ initialNodes: nodes, initialEdges: edges }))
    .not.toThrow();
});
```
